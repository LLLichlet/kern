# Kern 0.7.0 Release Checklist

This document defines the release gate for `0.7.0`.

The goal of `0.7.0` is not "add more features". The goal is to ship a healthier
compiler/toolchain boundary:

- explicit runtime/library configuration
- a real MIR-centered optimization pipeline
- stable CGU/LTO behavior
- a trustworthy `craft` release path
- no known high-frequency correctness bugs or compiler ICEs in the current
  language/toolchain surface

`0.7.0` should be tagged only when every item in the required sections is
complete.

## Release Criteria

`0.7.0` must satisfy all of these:

- `cargo test` passes at the workspace root.
- `craft build`, `craft test`, and `craft publish` behave consistently with the
  documented profile/runtime/LTO model.
- `kernc` runtime/library CLI semantics are explicit, orthogonal, and match the
  documentation.
- MIR is the compiler-owned mid-level optimization boundary for the current
  pipeline; no hidden fallback path should silently bypass that boundary.
- ThinLTO/full-LTO behavior is documented, test-covered, and stable enough to
  recommend as real release-profile machinery rather than as an experiment.
- There are no known reproducible ICEs on core language paths currently relied
  on by the toolchain and standard libraries.

## Required Before Tag

### 1. Runtime/Library Model Is Fully Closed

- Keep only the explicit axes:
  `runtime-entry`, `runtime-libc`, `library-bundle`.
- Do not reintroduce `runtime-provider` or other overlapping compatibility
  vocabulary.
- Ensure docs, CLI help, manifest handling, and implementation all describe the
  same model.
- Keep `base`/`sys` as explicit library roots rather than auto-imported
  preludes.
- Preserve the freestanding-first design: `no_std` behavior remains the default
  model, while hosted/runtime wiring stays explicit.

### 2. MIR Boundary Is Healthy

- MIR owns mid-level optimization work for the current compiler pipeline.
- `kernc_mir` owns MIR data structures, verification, summaries, and passes.
- `kernc_mir_lower` owns the staged lowering boundary from MAST into MIR.
- Codegen consumes MIR directly rather than depending on source-level fallback
  shapes.
- New optimizer work should target MIR first unless there is a clear reason to
  keep it elsewhere.

### 3. CGU/LTO Story Is Stable

- `--codegen-units`, `--lto none|full|thin`, and `--emit-llvm` stage behavior
  are documented and tested.
- Release-profile ThinLTO is the default optimized path in `craft`.
- Full LTO remains explicit rather than accidental.
- Non-LTO multi-object builds remain a first-class baseline.
- ThinLTO import/export planning and final-link behavior are stable enough that
  symbol/linkage behavior is predictable under real package builds.

### 4. `craft` Release Path Is Real

- `craft` release profiles must map cleanly onto `kernc` optimization/LTO
  settings.
- `craft publish` must remain a local release-readiness check, not an implicit
  state-mutating command.
- Release lockfile rules must be deterministic and enforced.
- Cross-package optimized builds must preserve the intended ThinLTO path instead
  of collapsing optimization inside each package independently.

### 5. Language Correctness And ICE Closure

- `let else` must be stable, including nested/high-frequency error-handling use.
- Anonymous aggregate coercions must lower correctly for both literal and
  non-literal values.
- Inline asm, SIMD, and other currently exercised low-level features must not
  regress on the MIR/codegen boundary.
- Any known reproducible compiler ICE in the current mainline surface should be
  fixed before release instead of documented away.

### 6. Documentation Matches Reality

- `docs/kernc.md` matches the actual CLI and behavior.
- `docs/craft.md` matches the actual profile/release/build model.
- `docs/compiler-optimization-architecture.md` matches the implemented MIR/LTO
  architecture closely enough to serve as a real design reference.
- Tutorial/workflow docs should not describe removed flags or obsolete runtime
  layering terminology.

## Strongly Recommended For 0.7.0

These are not as hard as the release gate above, but should be completed if
they do not destabilize the required work:

- add finer-grained `--emit-llvm` staging so IR inspection can target the
  specific pipeline layer under investigation
- finish cleanup of naming and crate boundaries where historical
  `flow`/`lower`/`mast` placement is still confusing
- review Windows/macOS runtime/linkage code paths for obvious drift while Linux
  remains the primary validation platform
- keep strengthening MIR verification so malformed lowering/codegen boundaries
  fail early and clearly

## Explicit Non-Blockers For 0.7.0

These matter, but they should not hold the release hostage:

- making the incubator JSON library "the final industrial implementation"
- exhaustive performance work on every third-party library
- adding new type-system features such as associated types unless they unlock a
  real missing capability
- broad standard-library expansion that is not required by the current
  toolchain or core ecosystem

The JSON library should continue to be used as a stress test for parser/codegen
quality, MIR optimization, SIMD support, allocator/container ergonomics, and
toolchain performance. It is an important workload, but not the definition of
`0.7.0` itself.

## Pre-Tag Checklist

Immediately before tagging `0.7.0`, verify all of the following:

- workspace `cargo test` passes
- targeted `craft` dev and release flows pass on representative packages
- release-profile ThinLTO builds succeed from `craft`
- current docs still match the shipped CLI and manifest semantics
- no known open ICE/regression is being consciously deferred inside the required
  release surface

If any of those fail, the release is not ready.
