# Kern 0.7.0 Language Health Draft

This document records a set of language-level redesign directions intended for
`0.7.0`.

These are not framed as convenience features or compatibility-preserving
extensions. They are health-driven language corrections derived from real
project pressure.

The core premise is simple:

- Kern prefers one correct model over two partially overlapping models.
- Kern rejects long-lived "transition syntax" that keeps unhealthy forms alive.
- If a foundational part of the language is wrong, it should be rebuilt
  cleanly rather than patched incrementally.
- A language feature should appear in source exactly as the language means it,
  not as library sugar pretending to be syntax.

This draft is intentionally opinionated. It is meant to be reviewed, corrected,
then used as the basis for implementation work.

## 1. Health Principle

For the features in this document, Kern should follow these rules:

- one semantic concept must have one canonical surface form
- language-level semantics must not be duplicated by public library spellings
- fundamental type, module, and pointer rules must be globally coherent
- compatibility with older formulations is not a goal when those formulations
  preserve an unhealthy model

The anti-goal is "ship a half-finished compromise that must be explained away
forever".

## 2. Associated Types

### 2.1 Position

Associated types are a real missing capability in Kern's trait system and
should be added as first-class trait items.

This should not be introduced as a partial mechanism. If added, it must be
supported through parsing, semantic checking, generic substitution, and
monomorphization.

### 2.2 Canonical Syntax

Trait declarations use semicolon-terminated associated type items:

```kern
type Add[Rhs] = trait {
    type Out;
    add: fn(Rhs) Out,
};
```

Key decisions:

- associated types end with `;`, not `,`
- methods should refer to associated types directly as `Out`, not `Self.Out`
- the direct name is preferred because Kern already omits `Self` in ordinary
  method usage, and reintroducing it here would make the surface less direct

### 2.3 Impl Form

Implementations define associated types directly inside the impl:

```kern
impl Vec2: Add[i32] {
    type Out = Vec2;
    fn add(rhs: i32) Out {
        ...
    }
}
```

Notes:

- this must follow Kern's existing impl syntax rather than importing a
  Rust-style surface
- Kern is value-oriented, so pointer impl targets such as
  `impl *mut Vec2: Add[i32]` remain part of the same orthogonal system
- associated type definitions should compose with the existing type language,
  including forms such as `type Out = ****i32;`

### 2.4 Constraints

From a health and orthogonality perspective, associated types should support:

- generic parameters on the associated type itself
- bounds or constraints on the associated type

These should not be artificially forbidden if the surrounding trait system can
express them coherently.

### 2.5 Explicit Non-Goals

- no default associated types

Kern does still need direct trait-object construction restrictions, but they
should be specified as simple compilation constraints in Kern's own terms
rather than by importing Rust's object-safety framing wholesale.

The intended direction is:

- associated types remain orthogonal
- trait-object formation rules remain explicit and easy to reason about
- the language stays firmly in a plain value/type model rather than drifting
  toward OOP semantics

## 3. Module Visibility

### 3.1 Position

The current direction of Kern's module visibility model is not considered
broken in principle. The goal is not to import Rust's visibility vocabulary
wholesale.

The only extension currently worth serious discussion is a clearer syntax for
upward-scoped visibility, similar in role to Rust's `pub(super)`.

### 3.2 Direction

One candidate direction is a concise form such as:

```kern
pub..name = ...;
```

This tighter spelling is preferable if it does not create parser ambiguity.

The intended role is still "visible to the parent scope" or an immediately
enclosing visibility boundary.

This remains open and should be discussed before implementation.

### 3.3 Constraints

Any visibility extension should preserve these properties:

- visibility remains easy to read at the declaration site
- path-based visibility does not become verbose or policy-heavy
- private implementation organization must stay easy to refactor

The goal is clarity, not feature count.

## 4. Pointer Redesign

### 4.1 Position

Pointers should remain expressive low-level values. Kern is not trying to turn
pointer use into a safe-only subsystem.

However, pointer construction rules should be cleaned up so the model is
healthier and easier to reason about.

### 4.2 Canonical Rules

The redesign direction is:

- remove direct pointer value initialization forms such as `*i32.{0}`
- pointers must come from a real pointer-producing operation
- raw address integers enter pointer space through `usize as ^T`
- object pointers and address pointers are separate families
- pointer arithmetic should no longer treat object pointers as address integers

The full model is specified in `docs/0.7.0-pointer-model.md`.

### 4.3 Non-Null by Default

The core semantic change is:

- `*T` is non-null
- `*mut T` is non-null
- `^T` is an address / volatile pointer family and may represent address `0`
- `^mut T` follows the same address / volatile model

Non-null is the one privileged property of object pointers in Kern's design.

### 4.4 Nullable Pointers

The only canonical nullable pointer form should be optional pointer types:

- `?*T`
- `?*mut T`

The only canonical null value should therefore be:

- `?*T.None`

There should not be any second null model.

User-facing explanation may still say "you can understand `?*T` as an optional
pointer", but this is only an intuition aid. The actual model is that `?T` is
a privileged builtin type family.

This also means:

- users may still define an ordinary enum named `Option`
- that user-defined enum is just a normal enum
- code such as `Option[?*T]` should remain valid as an expression of
  orthogonality, not as evidence that builtin `?T` is library sugar

`?^T` should not exist, because `^T` already includes raw address `0` and a
second nullable spelling would recreate overlap.

### 4.5 FFI Mapping

FFI pointer types should be chosen by semantics:

- non-null object pointer: `*T`
- nullable object pointer: `?*T`
- address / volatile pointer: `^T`

`?*T` must use the raw null-pointer ABI representation so nullable foreign
object pointers map directly and without an extra tag.

## 5. Builtin Optional and Result Types

### 5.1 Position

If Kern adopts `?T` and `T!E`, then those are not syntax sugar.

They are the canonical language forms for optional values and result-carrying
values.

Therefore:

- public language usage should not continue to expose `Option[T]`
- public language usage should not continue to expose `Result[T, E]`

Keeping both forms alive would violate Kern's health principle by creating two
surface models for the same semantics.

This does not forbid users from defining ordinary enums named `Option` or
`Result`. It only means those user-defined enums do not occupy the same
semantic tier as builtin `?T` and `T!E`.

### 5.2 Canonical Forms

Optional values:

- type form: `?T`
- constructors and matching remain enum-like:

```kern
let a = ?i32.{ Some: 10 };
let b = ?i32.None;
```

Result values:

- type form: `T!E`
- constructors and matching remain enum-like:

```kern
let ok = i32![]u8.{ Ok: 7 };
let err = i32![]u8.{ Err: "bad" };
```

### 5.3 Error and Empty Propagation

The propagation operators should be language-level:

- `.?` propagates `None`
- `.!` propagates `Err`

These are not library methods and not desugaring that should remain visible to
users as a second model. They are part of the language's direct control-flow
surface.

### 5.4 Nesting

One major benefit of the canonical forms is readable composition:

- `?i32`
- `i32![]u8`
- `?i32![]u8`

The exact precedence and associativity rules still need to be pinned down in
the parser and design document.

### 5.5 Relationship to Pointers

This model aligns directly with the pointer redesign:

- nullable pointer: `?*T`
- null value: `?*T.None`

The same omission and nesting rules should compose across `?T`, `T!E`, and
pointer-containing forms.

That gives the language one null model and one optionality model instead of
two unrelated mechanisms.

## 6. Current Decisions

At the current discussion stage, these points are treated as agreed:

- associated types should be added as real trait items
- associated type declarations end with `;`
- associated types are referenced directly by name such as `Out`
- default associated types should not exist
- associated types should remain orthogonal, including generic and constrained
  forms
- pointer direct-literal construction should be removed
- raw pointers remain expressive low-level values
- `usize as ^T` is the canonical raw-address entry operation
- `usize as *T` should not remain legal
- `usize as ?*T` should be legal and lower directly to the nullable object-pointer airlock
- non-null object pointers should become the default model
- the only canonical null model should be `?*T.None`
- `^T` is the raw address / volatile family and may represent address `0`
- `?^T` should not exist
- object-pointer arithmetic should not remain legal
- `?T` and `T!E` are canonical language forms, not sugar
- `Option[T]` and `Result[T, E]` should not remain as parallel public spellings
- `.?` and `.!` should be language operators

## 7. Open Questions

The remaining questions to settle before implementation are:

- exact syntax and semantics for upward-scoped visibility such as `pub..`
- exact parser precedence for `?T` and `T!E`
- exact typing rules for `.?` and `.!`
- exact trait-object construction restrictions in Kern terms
- whether trait-associated-type constraints need any staged implementation
  ordering even if the end model is fully orthogonal

## 8. Intended Outcome

If this redesign lands correctly, Kern should gain:

- a healthier trait system
- a cleaner pointer model
- one canonical optional/result surface
- a language that is easier to refactor without carrying historical duplicate
  forms forever

That is the real goal of this work.
