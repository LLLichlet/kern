# Kern 0.7.0 Pointer Model

This document fixes the pointer model targeted for `0.7.0`.

The goal is health, not compatibility with older or looser pointer rules.

Kern uses three distinct pointer families:

- `*T` / `*mut T`: object pointers
- `?*T` / `?*mut T`: nullable object pointers
- `^T` / `^mut T`: address / volatile pointers

These are not interchangeable spellings for the same thing. They represent
three different semantic tiers.

## 1. Object Pointers

`*T` and `*mut T` are object pointers.

Rules:

- they are non-null
- they model pointers to actual objects
- they are the normal pointer family used by ordinary Kern code
- they do not carry volatile access semantics

The language should never provide a direct path that can fabricate a zero
`*T`.

## 2. Nullable Object Pointers

`?*T` and `?*mut T` are the only canonical nullable object-pointer forms.

Rules:

- `?*T.None` is the only canonical null object-pointer value
- `?*T.{ Some: ptr }` holds a non-null `*T`
- there is no second null model for object pointers

This means Kern has exactly one object-pointer null model:

- non-null object pointer: `*T`
- nullable object pointer: `?*T`

## 3. Address / Volatile Pointers

`^T` and `^mut T` are address / volatile pointers.

They exist for low-level address-oriented programming:

- MMIO
- registers
- fixed hardware addresses
- volatile memory locations
- low-level FFI boundaries that are fundamentally address-like rather than
  object-like

Rules:

- `^T` may represent address `0`
- `^T` is the only pointer family that may be constructed directly from raw
  address integers
- `^T` is not an object pointer
- `^T` is the right place for raw address manipulation and volatile access

## 4. `volatile` Is an Access Property

The `volatile` meaning attached to `^T` applies to memory access performed
through that pointer family.

It is not a sticky property of the underlying address value itself.

Therefore:

- `usize as ^T` constructs an address/volatile pointer value
- operations performed through `^T` use volatile memory semantics
- converting `^T` into `?*T` or extracting `*T` from that optional form strips
  volatile access semantics

This is required so `^T` can act as the explicit address gateway without
poisoning later object-pointer optimization.

## 5. Canonical Cast Model

The intended cast rules are:

- `*T as usize`: allowed
- `^T as usize`: allowed
- `usize as ^T`: allowed
- `usize as *T`: forbidden
- `usize as ?*T`: allowed
- `*T as ^T`: allowed
- `^T as ?*T`: allowed
- `^T as *T`: forbidden

The nullable object-pointer airlock may be entered either from an integer
address or from an address pointer:

- address `0` becomes `None`
- any non-zero address becomes `Some`

So both of these are valid and mean the same null-filtering step:

- `usize as ?*T`
- `^T as ?*T`

After that conversion, ordinary optional handling decides whether a real object
pointer exists.

## 6. Why `?^T` Does Not Exist

`?^T` would duplicate the meaning of address-zero in a second surface form.

That would recreate the same kind of unhealthy overlap Kern is trying to
remove elsewhere.

So the model is:

- `^T` already covers raw address `0`
- `?*T` covers nullable object pointers
- `?^T` should not exist

## 7. Pointer Arithmetic

Object pointers are not address-integer values.

Therefore Kern should not support direct `+` / `-` arithmetic on `*T` or
`*mut T`.

Rules:

- `*T` pointer arithmetic is forbidden
- address arithmetic belongs to `^T`, `^mut T`, or integer address values such
  as `usize`
- object navigation should instead use indexing, slices, or explicit lower-level
  address code when that is truly what the programmer means

This avoids wraparound paths that could reintroduce zero into the `*T` world.

## 8. FFI Mapping

FFI pointers must be declared by their real semantics, not by C's historically
loose spelling.

Use:

- `*T` for non-null object pointers
- `?*T` for nullable object pointers
- `^T` for address-like / volatile pointers

Examples:

```kern
extern fn memcpy(dst: *mut u8, src: *u8, n: usize) *mut u8;
extern fn getenv(name: *u8) ?*u8;
extern fn mmio_base() ^mut u32;
```

FFI contracts:

- if an extern signature says `*T`, the foreign side must not pass or return
  null
- if null is part of the foreign contract, the Kern type must be `?*T`
- address-like hardware or volatile locations should use `^T`

## 9. ABI of `?*T`

`?*T` must use the null-pointer representation directly.

Rules:

- `size_of[?*T] == size_of[*T]`
- `align_of[?*T] == align_of[*T]`
- `?*T.None` is encoded as address `0`
- `?*T.{ Some: ptr }` is encoded as the pointer value of `ptr`

`?*T` is therefore not a separately tagged enum representation at the ABI
boundary.

This rule is required for clean FFI and for keeping nullable object pointers as
the canonical null model.

## 10. Safety Boundary

This model eliminates one major unhealthy hole:

- zero can no longer silently enter `*T`

It does not claim to prove that every non-zero address is a valid object.

The following can still be wrong if the programmer lies at the boundary:

- alignment
- lifetime
- provenance
- aliasing
- object validity

That is acceptable for a low-level language. The health goal here is to place
the danger behind explicit, minimal, semantically clear gateways.

## 11. Summary

The final model is:

- `*T`: non-null object pointer
- `?*T`: nullable object pointer
- `^T`: raw address / volatile pointer
- raw integers enter pointer space only through `^T`
- nullable object pointers use only `?*T`
- `?^T` does not exist
- object-pointer arithmetic is forbidden

That is the `0.7.0` direction.
