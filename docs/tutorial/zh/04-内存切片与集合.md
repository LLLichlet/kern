# 04. 内存、切片与集合

[English](../en/04-memory-slices-and-collections.md) | 简体中文

这一章开始进入 Kern 比较底层的一面：数组、切片、指针、显式分配和作用域清理。它们不是彼此独立的功能点，而是同一套规则在不同地方的展开：值有类型，存储位置决定是否可写，库类型通常把资源管理交给调用者明确安排。

## 数组与切片

固定长度数组写成 `[N]T`，其中 `N` 是元素个数，`T` 是元素类型：

```kern
let mut values = [5]i32.{ 1, 2, 3, 4, 5 };
values.[0] = 9;
```

`[5]i32.{ ... }` 仍然是前面章节讲过的“类型提供者 + 初始化体”。数组元素的类型已经由外层 `[5]i32` 给出，所以元素里通常不需要再写 `1i32`、`2i32`。

数组也支持重复初始化：

```kern
let zeroes = [4]u8.{ 0; 4 };
```

如果数组元素本身还是聚合值，内层也可以继续使用上下文初始化体：

```kern
let matrix = [2][3]i32.{
    .{ 1, 2, 3 },
    .{ 4, 5, 6 },
};
```

这里外层类型说明了每个元素都是 `[3]i32`，所以内层可以写 `.{ 1, 2, 3 }`。

数组长度也可以让编译器从初始化元素个数推导：

```kern
let inferred = [_]i32.{ 1, 2, 3 };
```

`[_]i32` 仍然是数组类型，只是长度由初始化体决定。它适合写查表数据、测试输入、短的固定序列。

需要直接得到切片时，也可以把切片类型作为类型提供者：

```kern
let direct = &[i32].{ 1, 2, 3 };
```

这会先形成后备数组值，再得到指向它的只读切片。切片仍然不拥有元素；这只是把“创建数组并取切片”写成一个初始化表达式。

切片是胖指针，携带数据指针和长度。它不拥有元素，只是看向一段连续存储：

```kern
let middle = values.&[1...4];
let head = values..&[0...3];
```

`values.&[1...4]` 产生只读切片 `&[i32]`，范围是 `[1, 4)`。`values..&[0...3]` 产生可写切片 `&mut [i32]`。

范围可以省略左端或右端：

```kern
let prefix = values.&[...3];
let tail = values.&[2...];
let whole = values.&[...];
let inclusive = values.&[1..=3];
```

`...3` 表示从开头到 `3` 之前，`2...` 表示从 `2` 到末尾，`...` 表示整个范围，`1..=3` 表示包含右端点 `3`。slice 范围的端点是内存偏移，所以端点类型必须是 `usize`；无后缀数字通常会从 slice 语境推断为 `usize`。

这些范围表达式本身是内置 range 值。放在 slice 方括号里时，它们必须是编译器拥有的 `SliceBounds` 形状，比如 `usize...usize`、`...usize`、`usize...`、`usize..=usize` 或 `...`。`SliceBounds` 只是“能作为 slice 边界”的标记；用户代码不能实现它来发明新的切片语义。

这里的 `mut` 不写成 `[5]mut i32`，因为 Kern 的可变性属于存储位置，不属于元素类型。数组和结构体描述的是数据形状；某个数组值能不能被改，取决于它所在的存储是否可写：

```kern
let fixed = [3]i32.{ 1, 2, 3 };
let fixed_view = fixed.&[...];

let mut editable = [3]i32.{ 1, 2, 3 };
let editable_view = editable..&[...];
editable_view.[0] = 9;
```

`fixed` 没有用 `let mut` 声明，所以只能取只读视图。`editable` 的本地存储可写，所以可以取 `&mut [i32]` 并通过切片修改元素。

长度可以用语言原语 `#` 取得；日常代码里更推荐用标准库方法 `.len()`，语义更直接：

```kern
fn sum(items: &[i32]) i32 {
    let mut total = 0;
    let mut i = 0;
    while (i < items.len()) {
        total += items.[i];
        i += 1;
    }
    return total;
}
```

`items.@len()` 是编译器拥有的切片长度原语，不依赖标准库。`items.len()` 则是普通的标准库包装，日常代码可以优先使用这种写法。

## 迭代器

`base.coll` 为内置 range 值提供 iterator 实现，同时提供 slice iterator 和常见消费方法。最直接的用法是 `for`：

```kern
let mut total = 0;
for (i: 1...4) {
    total += i * i;
}
```

需要反向遍历时，对已有 range 值调用 `.rev()`：

```kern
let mut descending = 0;
for (i: (1...4).rev()) {
    descending = descending * 10 + i;
}
```

这会依次访问 `3`、`2`、`1`。这里的括号是有意义的：`a...b.rev()` 表示 `a...(b.rev())`，不是 `(a...b).rev()`。

切片可以调用 `.iter()`。数组在方法调用和参数传递时可以自然转成切片，所以遍历整个数组通常可以直接写：

```kern
let values = [3]i32.{ 1, 2, 3 };
let mut total = 0;

for (item: values.iter()) {
    total += item;
}
```

这行调用的是 `&[i32]` 上的 `.iter()`；`values` 是 `[3]i32`，在这里自然衰减成只读切片。需要强调切片边界，或者只遍历一部分时，再显式写出切片：

```kern
for (item: values.&[1...].iter()) {
    total += item;
}
```

迭代器在 Kern 里是显式状态值。`for` 循环可以理解成下面这种形状：

```kern
let mut iter = 1...4;
while (true) {
    let .{ Some: i } = iter..&.next() else break;
    total += i * i;
}
```

也就是说，`for` 会保存一个可变的迭代器状态，并反复调用 `next()`。`next()` 返回 `?Item`：有下一个元素时返回 `Some(item)`，结束时返回 `None`。

`continue` 和 `break` 仍然按普通循环控制流工作：

```kern
for (i: 0...10) {
    if (i == 2) continue;
    if (i == 7) break;
    total += i;
}
```

自己实现迭代器时，核心是给可变接收者实现 `Iterator` trait：

```kern
use base.coll.Iterator;

struct CountTo {
    current: usize,
    limit: usize,
};

impl &mut CountTo : Iterator {
    type Item = usize;

    pub fn next() ?Item {
        if (self.current >= self.limit) {
            return .None;
        }

        let item = self.current;
        self.current += 1;
        return .{ Some: item };
    }
}

fn sum_small() usize {
    let mut total = 0usize;
    for (i: CountTo.{ current: 0, limit: 4 }) {
        total += i;
    }
    return total;
}
```

为什么实现写在 `&mut CountTo` 上？因为迭代会推进内部状态。读一个元素之后，`current` 必须改变，所以 `next()` 需要可写接收者。

迭代器也可以直接使用消费方法。因为这些方法会消耗迭代器状态，通常先把迭代器放进可变本地存储，再用 `..&` 调用：

```kern
let values = [5]i32.{ 1, 2, 3, 4, 5 };
let mut iter = values.iter();
let sum = iter..&.fold[i32](0, [](accum: i32, value: i32) i32 {
    return accum + value;
});
```

如果要遍历并修改元素，需要先得到可写切片：

```kern
let mut values = [3]i32.{ 1, 2, 3 };
for (item: values..&[...].iter()) {
    item.* += 1;
}
```

`values..&[...]` 是 `&mut [i32]`，它的 iterator 产出 `&mut i32`，所以循环体里用 `item.*` 写回元素。

## 指针模型

Kern 当前有两组常用指针：

- `&T` / `&mut T`：普通对象指针。
- `^T` / `^mut T`：地址/volatile 指针，适合 MMIO 和固定地址访问。

普通代码大多使用 `&T` 和 `&mut T`。取地址时，Kern 把只读借用和可写借用写成两个不同操作：

```kern
let mut value = 10i32;

let read_ptr = value.&;
let write_ptr = value..&;

let current = read_ptr.*;
write_ptr.* = current + 1;
```

`value.&` 得到 `&i32`，表示“可以读这个 `i32`”。`value..&` 得到 `&mut i32`，表示“可以通过这个指针修改这个 `i32`”。二者分开写，是为了让读权限和写权限在源码里保持明确。

`.*` 是显式解引用。Kern 不做自动解引用，所以读写指针目标都要写出来：

```kern
let before = read_ptr.*;
write_ptr.* = before + 1;
```

可写指针通常也可以使用只读接收者上的方法。库作者会把不会修改对象的方法放在 `impl &T` 上，把会修改对象的方法放在 `impl &mut T` 上；调用者拿着 `&mut T` 时，仍然可以调用只读方法。这让 API 可以把“哪些操作会改变值”放在方法接收者上表达。

## `defer` 与显式释放

Kern 没有垃圾回收，也没有隐藏的析构策略。需要在作用域结束时执行清理代码时，用 `defer`：

```kern
{
    acquire();
    defer release();

    work();
}
```

`defer` 在当前块退出时执行，顺序是后注册的先执行。正常走到块末尾、`return`、`break`、`continue` 离开这个块时，都要先执行这个块里已经注册的 `defer`。

如果块本身要产生一个值，Kern 会先算出块的结果，再执行 `defer`，最后把结果交给外层。这保证了执行顺序是明确的，但也意味着不要返回指向同一块内即将释放资源的指针。

## 标准库容器的分配模式

Kern 标准库的数据结构 API 一般需要显式 allocator。更准确地说，这不是语言语法强制的规则，而是标准库的设计：`List[T]`、`String` 这类容器不把 allocator 存进自己内部。每次可能分配、扩容或释放的操作，都由调用者显式传入 allocator。

典型写法是先取得一个分配器，再在当前作用域里创建容器值，随后立刻安排 `defer`：

```kern
use base.coll.{list, string};
use base.mem.alloc.gpa;
use std.mem.page;

let page = page()..&;
let gpa = gpa().on(page)..&;

let numbers = list[i32]()..&;
defer numbers.deinit(gpa);

if (!numbers.push(gpa, 3)) return;
if (!numbers.push(gpa, 1)) return;
if (!numbers.push(gpa, 2)) return;

let text = string()..&;
defer text.deinit(gpa);

if (!text.push_str(gpa, "Hello")) return;
if (!text.push_str(gpa, ", Kern")) return;
```

这里有几个点要连起来看：

- `list[i32]()` 创建的是空的 `List[i32]` 值；这个小的容器头本身可以放在当前作用域里。
- `..&` 取得这个容器值的可写接收者，后续 `push`、`deinit` 都通过它调用。
- `push(gpa, value)` 可能分配，所以 allocator 是显式参数。
- `deinit(gpa)` 释放容器持有的后备缓冲区；它不会神秘地知道应该用哪个 allocator。
- `numbers.len()` 这样的只读方法仍然可以从 `&mut List[i32]` 调用，因为读方法由库放在只读接收者上。

这种模式和 C++ 的 RAII 不一样：释放不会自动绑定到对象生命周期上；但也不是手写一堆分散的 cleanup label。Kern 的常见写法是“资源值在当前作用域里，清理动作紧跟创建点用 `defer` 注册”，让资源的获得和释放在源码上靠得很近。

更完整的容器示例见 [`examples/collections.rn`](../../../examples/collections.rn) 和 [`examples/string.rn`](../../../examples/string.rn)。标准库容器的内部设计会放到后续更深入的库章节里讲，这一章先掌握调用侧的模式。

## 文件 I/O

`std.fs` 提供面向用户的文件系统便利层。路径通常由字节字符串调用 `.path()` 得到。这个例子保留了显式 allocator、`match` 错误处理和 `defer` 清理：

```kern
use base.mem.alloc.gpa;
use std.fs;
use std.io;
use std.mem.page;

const SUCCESS = 0i32;
const FAILURE = 1i32;

fn main() i32 {
    let page = page()..&;
    let gpa = gpa().on(page)..&;

    let path = ".craft/example.txt".path();
    _ = path.remove_file_if_exists(gpa);

    let written = match (path.write_all_atomic(gpa, "kern examples\n")) {
        .{ Ok: value } => value,
        .{ Err: _ } => return FAILURE,
    };

    let mut text = match (path.read_to_string(gpa)) {
        .{ Ok: value } => value,
        .{ Err: _ } => return FAILURE,
    };
    defer text..&.deinit(gpa);

    "wrote {} bytes: {}".fmt(.{written, text.&.as_str()}).println();
    _ = path.remove_file_if_exists(gpa);
    return SUCCESS;
}
```

`read_to_string` 返回拥有后备缓冲区的字符串值。例子里先用 `let mut text` 接住这个拥有值，再用 `defer text..&.deinit(gpa);` 绑定释放动作。打印时只需要读取内容，所以用 `text.&.as_str()` 取得只读接收者。

真实 CLI 或 app 项目里，通常会让 `main` 只负责搭建 allocator、调用业务函数、把错误映射成退出码；主流程可以放进 `fn app(...) void!MyError` 或类似函数中。这样错误类型、退出码和资源清理会更有结构。本教程后续进入实战项目时再展开这个组织方式。

## `const` 与 `static`

本章最后补上两个和存储身份密切相关的声明。

`const` 是编译期名字，不创建运行时对象、地址或链接符号：

```kern
const SUCCESS = 0i32;
const PAGE_SIZE = 4096usize;
```

使用 `SUCCESS` 就像在使用它代表的编译期值。适合放退出码、位标志、数组长度、平台常量等。

`static` 创建真正有存储身份的全局对象；需要全局可写存储时写 `static mut`：

```kern
static VERSION = [4]u8.{ b'k', b'e', b'r', b'n' };
static mut BOOT_COUNT = 0usize;
```

`static` 也可以写在块里，位置和 `let` 类似，但它创建的仍然是静态存储：

```kern
fn default_name() &[u8] {
    static name = "kern";
    return name;
}
```

这在需要返回稳定字节数据时很有用。字符串字面量本身是数组值；放进 `static` 后，`name` 有固定存储身份，返回 `&[u8]` 不会指向已经离开的局部临时值。

能用 `const` 表达的常量，优先用 `const`。只有确实需要固定地址、全局对象身份、与外部 ABI 交互，或者需要长期存在的全局状态时，才使用 `static` / `static mut`。
