# 05. 模块、包与库分层

[English](../en/05-modules-packages-and-libraries.md) | 简体中文

前面几章主要在讲语法。到了真实项目里，源码会被拆成多个文件，构建系统也需要知道入口、依赖和 runtime 选择。本章先建立几个容易混淆的层次：

- 模块是 Kern 语言里的命名空间和源码组织单位。
- 包是 `Craft.toml` 描述的构建单位，一个包可以有库、可执行文件、示例和测试等 target。
- workspace 是一组包的共同工作区，用来共享依赖声明、锁文件和构建状态。
- 官方库按 freestanding-first 的方式分层，并通过 runtime `bundle` 选择接入常用 root alias。

## 一个小包长什么样

先看一个普通可执行包的目录：

```text
hello/
  Craft.toml
  src/
    main.rn
    math.rn
```

`craft init` 生成的普通命令行包会接近下面这个最小形态：

```toml
[package]
name = "hello"
version = "0.1.0"
kern = "0.7.5"

[[bin]]
name = "hello"
root = "src/main.rn"
```

这里已经足够构建一个 hosted 可执行程序。`[[bin]]` 声明一个 binary target，`root = "src/main.rn"` 表示这个 target 的根模块是 `src/main.rn`。

一般应用不需要写 `[runtime]`。对 binary、example 和 test target，`craft` 默认使用工具链 runtime startup、不隐式链接 libc，并接入 `std` bundle。只有你在写 kernel、freestanding 程序、自定义启动入口，或者需要明确改变库 bundle 时，才需要把 runtime 配置写出来：

```toml
[package]
name = "hello"
version = "0.1.0"
kern = "0.7.5"

[runtime]
entry = "rt"
libc = false
bundle = "std"

[[bin]]
name = "hello"
root = "src/main.rn"
```

`src/main.rn` 是这个 binary target 的根模块。它可以声明子模块：

```kern
mod math;

use std.io;

fn main() i32 {
    let value = math.square(7);
    "square = {}".fmt(.{value}).println();
    return 0;
}
```

`src/math.rn` 是同级文件，被 `mod math;` 接入：

```kern
pub/ fn square(value: i32) i32 {
    return value * value;
}
```

这里的 `pub/` 表示包内可见。`main.rn` 和 `math.rn` 在同一个包里，所以 `main` 可以调用 `math.square`；包外代码不能把它当作公开 API 使用。

## `mod` 声明模块

Kern 的模块树由源码显式声明，不由文件系统自动扫描。一个文件放在目录里，并不会自动参与编译；它必须被某个已接入模块用 `mod` 声明出来。

`mod name;` 在当前模块下面声明一个子模块。模块声明本身也有可见性：默认只在当前模块局部可见；需要让包内其他模块通过路径访问它时，可以写 `pub/ mod name;`；需要公开到包外时才写 `pub mod name;`。

编译器会按固定规则找模块文件：

- 如果旁边有 `name.rn`，它是文件模块。
- 如果旁边有 `name/` 目录，目录模块的入口是 `name/init.rn`。

所以 `init.rn` 不是随便起名的普通文件，而是目录模块的入口文件。它通常负责声明该目录下继续拆分出来的子模块，并提供这个目录对外暴露的 API。

```text
src/
  main.rn
  parse/
    init.rn
    token.rn
```

`src/main.rn`：

```kern
mod parse;

fn main() i32 {
    return parse.run("123");
}
```

`src/parse/init.rn`：

```kern
mod token;

pub/ fn run(text: &[u8]) i32 {
    return token.count_digits(text);
}
```

`src/parse/token.rn`：

```kern
pub.. fn count_digits(text: &[u8]) i32 {
    let mut count = 0;
    for (byte: text.iter()) {
        if (byte >= b'0' and byte <= b'9') {
            count += 1;
        }
    }
    return count;
}
```

这里 `token` 是 `parse` 的子模块。`pub..` 表示父模块树可见，所以 `parse/init.rn` 能调用 `token.count_digits`，但它不会变成整个包的公开 API。

这和 C/C++ 的 `#include` 很不一样：`mod` 不是文本粘贴，也不需要头文件或前置声明。Kern 会收集模块树里的声明，再做多阶段解析，所以跨模块的类型引用可以直接写出来。

它也和 Rust 的直觉不完全一样。Kern 的路径根更明确，可见性边界也更细：包内可见用 `pub/`，父模块树可见用 `pub..`，真正公开给包外才用 `pub`。

## `use` 导入名字

Kern 没有自动 prelude。要用什么模块或名字，就显式导入什么：

```kern
use std.io;
use base.coll.{List, list, range};
use base.mem.alloc.gpa;
use std.mem.page;
```

`use std.io;` 把模块名 `io` 带入当前作用域。`use base.coll.{List, list, range};` 从同一个模块里导入多个名字。

导入只改变当前文件里怎么写名字，不会改变模块的可见性，也不会把依赖自动转发给别的模块。一个文件需要什么，就在这个文件里写对应的 `use`。

相对路径也很常见。标准库源码里经常能看到：

```kern
use ..types.{Error, Path};
```

`..` 表示父模块，适合子模块之间引用同一个父模块下的定义。`/` 表示当前包的模块根，接近 Rust 里的 `crate::`：

```kern
use /parse;

fn run_from_root(text: &[u8]) i32 {
    return /parse.run(text);
}
```

这些锚点不只用于 `use`，也可以直接出现在表达式和类型里：

```kern
fn count_from_parent(text: &[u8]) i32 {
    return ..math.square(token.count_digits(text));
}
```

上面这个例子假设父模块里有一个包内可见的 `math` 子模块，也就是根模块里写过 `pub/ mod math;`。路径能不能解析，不只取决于文件位置，也取决于沿途模块和名字的可见性。

项目内部路径可以按边界选择写法：跨官方库边界时使用 `base.coll`、`std.fs` 这样的 root alias；在同一个包内部整理模块时使用 `/`；在同一个目录树的子模块之间引用父级内容时使用 `..`。

## 可见性边界

先记住三个最常用的可见性：

- `pub`：包外可见，是真正公开 API。
- `pub/`：包内可见。
- `pub..`：父模块树可见。

默认不写可见性时，定义只在当前模块局部可见。

```kern
fn helper() i32 {
    return 1;
}

pub.. fn parse_local(text: &[u8]) i32 {
    return helper();
}

pub/ fn parse_package(text: &[u8]) i32 {
    return parse_local(text);
}

pub fn parse_public(text: &[u8]) i32 {
    return parse_package(text);
}
```

不要把所有东西都写成 `pub`。如果一个函数只是给同一个包内部使用，用 `pub/`；如果只是给父模块组织内部使用，用 `pub..`。这比在 `init.rn` 里写大量转发函数更直接，也更能表达真实边界。

结构体字段也有自己的可见性。命名结构体的字段默认只在定义它的模块里可访问；如果调用方需要直接读写字段，字段也要标成 `pub`：

```kern
pub struct Span {
    pub start: usize,
    pub end: usize,
};
```

这让 API 边界更清楚：你可以公开一个类型，但仍然把字段藏起来，只提供构造函数或方法。匿名结构体和匿名 union 没有独立的定义模块，字段不能声明 `pub`。

## 包和目标

`Craft.toml` 描述一个包如何被 `craft` 构建。最小可执行包通常只需要 `[package]` 和一个 `[[bin]]`；库包、示例和测试才需要对应 target。

库包：

```toml
[package]
name = "mylib"
version = "0.1.0"
kern = "0.7.5"

[lib]
root = "src/lib.rn"
```

`[lib]` 没有 `name` 字段；库 target 的名字就是包名。库 target 默认没有 runtime startup，因为它不是最终可执行文件。

可执行目标：

```toml
[[bin]]
name = "tool"
root = "src/main.rn"
```

`[[bin]]` 是数组表，所以一个包可以有多个可执行 target。普通应用只有一个可执行文件时，`craft init` 已经会写好这一段。

示例和测试目标：

```toml
[example]
roots = [
    "examples/smoke.rn",
    "examples/demo.rn",
]

[test]
roots = ["tests/smoke.rn"]
```

`[example]` 和 `[test]` 使用 `roots` 数组。每个 root 文件会成为一个独立 target，target 名字来自文件名，例如 `examples/basics.rn` 对应 `--example basics`。

仓库里的 [`examples/Craft.toml`](../../../examples/Craft.toml) 就是一个只声明 example roots 的包。运行一个示例：

```sh
craft run -p examples --example basics
```

构建全部示例：

```sh
craft build -p examples --examples
```

`craft` 负责选择包和 target、解析 manifest、维护 `Craft.lock`、安排构建动作。`kernc` 负责更底层的“把这些明确输入编译/链接成产物”。日常项目优先用 `craft`。

## 包、模块和 workspace 的区别

如果你来自 Python，可以把 Kern 的“模块”理解成语言层面的命名空间，但不要把它等同于 Python package：Kern 模块必须由 `mod` 明确接入，文件不会因为存在就自动可导入。

如果你来自 C/C++，可以把 `Craft.toml` 理解成项目的构建描述，把 `mod` 理解成源码结构声明。Kern 没有头文件包含模型；需要额外的 C 文件、链接脚本、生成文件或目标定制时，才进入 `build.rn`。需要更早、更全局地影响 package plan 时，才考虑 `craft.rn`。普通包先不用碰它们。

这三个词的边界可以这样记：

- module：语言里的源码组织单位，靠 `mod` 和 `use` 工作。
- package：一个 `Craft.toml` 描述的构建单位，里面可以有多个 target。
- workspace：多个 package 的命名空间根，用来共享依赖、锁文件、`.craft/` 构建状态，并声明哪些成员包导出给外部用户。

`craft init` 创建的是单包项目。workspace 是另一种根形态：根
`Craft.toml` 写 `[workspace]`，不写 `[package]`；真正能构建的包放在
member 目录里。

```toml
[workspace]
name = "json-kern"
members = [
    "json",
    "json-test",
    "json-bench",
]

[workspace.exports]
json = { member = "json" }

[workspace.package]
version = "0.1.0"
kern = "0.7.5"
license = "MIT"
authors = ["Example <dev@example.com>"]
readme = "README.md"
repository = "https://example.com/json-kern.git"

[workspace.dependencies]
json = { path = "json" }
```

成员包仍然拥有自己的包名和 target：

```toml
[package]
name = "json"
publish = true

[lib]
root = "src/lib.rn"
```

`[workspace.package]` 用来集中共享元数据。`version` 和 `kern` 会在成员省略时作为成员默认值；`description`、`license`、`authors`、`readme`、`repository` 会作为发布校验的共享默认值；`homepage` 和 `documentation` 是可接受的共享元数据字段。`publish` 不继承；每个成员包自己声明是否发布。

`[workspace.exports]` 是外部命名空间。workspace 可以包含测试、benchmark、代码生成工具等内部包；只有显式 export 的成员包会成为外部依赖可选的包。

## 依赖

包依赖写在 `Craft.toml` 的 `[dependencies]` 里。仓库内部常见的本地依赖是 path 依赖：

```toml
[dependencies]
mkiso = { path = "../limine-mkiso", export = "limine-mkiso" }
```

依赖一个外部 workspace 时，如果本地依赖名和导出名相同，可以直接写：

```toml
[dependencies]
json = { git = "https://example.com/json-kern.git", tag = "v0.1.0" }
```

如果本地想用另一个依赖名，就用 `export` 指定要选哪个导出包：

```toml
[dependencies]
kern_json = { git = "https://example.com/json-kern.git", tag = "v0.1.0", export = "json" }
```

workspace 里也可以复用根 `Craft.toml` 定义的依赖：

```toml
[workspace.dependencies]
base = { path = "base" }
json = { path = "json" }

[dependencies]
base = { workspace = true }
json = { workspace = true }
```

一个依赖进入包图之后，源码里仍然要用 `use` 明确导入需要的模块或名字。manifest 解决“这个包依赖谁”，`use` 解决“这个文件要用哪个名字”。

## 官方库分层

Kern 的官方库按 freestanding-first 的方式组织。它们不是一整块必须跟着编译器隐式进入程序的标准库，而是分层的 Kern 包；`craft` 根据 runtime `bundle` 把常用层接成 root alias。这种设计让普通 hosted 程序可以直接用 `std`，也让内核、bootloader、裸机工具和自定义 runtime 能只拿自己需要的层。

当前主要分层是：

- `base`：freestanding 基础库。包含比较、数字、指针和布局工具、allocator trait、集合、字符串、同步原语、通用 IO trait、测试断言等。
- `rt`：启动入口和最小 runtime glue。
- `std`：面向用户的高层库，建立在 `base` 上，hosted 实现在 `std.host` 内部。

`bundle = "std"` 会让 `base`、`std` 等官方 root alias 在普通 hosted 项目里可用。`bundle = "base"` 更适合 freestanding 或更底层的包。`bundle = "none"` 则表示不接入这些官方 alias。

重要的是：Kern 不把 libc 当成语言和标准库的基础设施。普通 target 默认 `libc = false`，`std` 的 hosted 能力通过内部 `std.host` 实现进入；libc 只是需要时才选择的外部 ABI。这样可以让静态链接的简单程序保持很薄，也让 freestanding 场景不用先绕过一整套 hosted 假设。更完整背景见 [`runtime-architecture.md`](../../runtime-architecture.md)。

## runtime 选择

普通命令行程序通常使用：

```toml
[runtime]
entry = "rt"
libc = false
bundle = "std"
```

含义是：

- `entry = "rt"`：使用工具链提供的 startup，并要求根模块里有合法 `main`。
- `libc = false`：不隐式链接 libc。
- `bundle = "std"`：接入官方标准库 root aliases。

freestanding 包通常会更显式：

```toml
[runtime]
entry = "none"
libc = false
bundle = "base"
```

这表示包自己拥有启动入口，不使用 runtime startup，只接入基础库层。本教程下一章会专门讲 freestanding 和 runtime。

## 什么时候看标准库源码

Kern 的标准库刻意保持显式、可读和低策略。学习语言时，源码本身就是很好的教程材料：

- [`library/base/option.rn`](../../../library/base/option.rn)：`?T` 的常用方法。
- [`library/base/result.rn`](../../../library/base/result.rn)：`T!E` 的常用方法。
- [`library/base/coll/iter.rn`](../../../library/base/coll/iter.rn)：iterator trait 与 `for` 背后的模型。
- [`library/base/coll/list_impl/init.rn`](../../../library/base/coll/list_impl/init.rn)：显式分配的 growable list。
- [`library/base/coll/string_impl/init.rn`](../../../library/base/coll/string_impl/init.rn)：`String` 如何建立在 `List[u8]` 上。
- [`library/std/io/init.rn`](../../../library/std/io/init.rn)：`println`、`Printable` 和标准流。
- [`library/std/fs/`](../../../library/std/fs/)：文件系统 API 如何通过 `std.host` 工作。

这些源码也体现了当前仓库风格：让上下文完成常规推断，但在 ABI、位宽、指针、可见性和低层边界处保持明确。
